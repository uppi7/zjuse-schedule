## 需求

新增"批量导入教室"接口（Excel / CSV 文件上传），让管理员一次导入大量教室；同时复审现有 5 个 CRUD 接口，确保对 `Classroom` model 的字段（特别是 `available_time`、`building`）支持完整、读写无丢失。

## 预期成果

- 新增 `POST /api/v1/classrooms/batch-import`：
  - 接受 `multipart/form-data` 上传 Excel 或 CSV。
  - 解析 + 行级校验；返回 `{success: int, failed: list[{row, code, error}]}` 结构（用 ApiResponse 包裹）。
  - 已存在 `code` 的教室默认跳过；可加 `?overwrite=true` 覆盖更新。
  - 仅 ADMIN（`require_admin`）。
  - 整体事务包裹：解析阶段失败回滚，导入阶段失败统计而非整体回滚（具体策略自定，写进 PR 描述）。
- 现有 CRUD 与 model 对齐：
  - `ClassroomUpdate` 当前缺 `building` 字段，补上。
  - `available_time` 非空场景下"创建 → 查询 → 更新 → 查询"全程数据无丢失、无类型错乱。
  - 现有 5 个端点全部走 `ApiResponse` + `BizException` 规范（已抽查 OK，本卡顺手再确认）。
- 修现存 `tests/unit/test_classrooms.py` 已坏的两处断言（写卡时发现的）：
  - `test_create_classroom` 发送 `code: "101"` 却断言 `data["code"] == "A101"`——必然失败
  - `test_create_classroom_duplicate_code` 期望 HTTP 409，但 service 抛 `BizException(GENERAL_ERROR, ...)`，按全局 handler 渲染为 HTTP 200 + `code=2000`——必然失败
  - 改成符合 ApiResponse 规范的断言：HTTP 200 + body `code===0`（成功）或 `code≠0` 且业务码符合预期
- 单测覆盖：
  - 批量导入：正常路径、部分行失败、文件格式错误、重复 code 的 skip/overwrite 两种行为。
  - `available_time` 非空 roundtrip（create→get→update→get）字段值一致。
  - `building` 字段可更新。
- Swagger 描述：endpoint summary/description 说清接受的文件列名和 `available_time` 在文件里的表达方式。

## 注意点

- 文件解析建议 `openpyxl`（Excel）+ `csv`（标准库）；新加依赖记得写进 `requirements.txt`。
- `available_time` 在 Excel/CSV 里的表达约定一列字符串（如 `"1-1,1-2,2-3"` 表示星期-节次对）；列定义写在 endpoint description 里让前端/测试都能看。
- 上传文件大小：`UploadFile` 默认无上限，需手动 check（如 ≤ 5 MB），超限抛 `BizException(BizCode.VALIDATION_ERROR, ...)`。
- 行级失败不要让整批回滚——记录失败行返回给调用方，让管理员针对性修。
- 前端上传 UI 当前在 F1 卡里被标了 TODO 不做；本卡上线后由前端工程师决定单开 issue 还是塞进 F1 的下个迭代——本卡不管前端。
