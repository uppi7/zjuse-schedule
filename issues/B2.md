## 需求

`app/services/teacher_preference_service.py` 里有 6 处 `NotImplementedError`：5 个对应路由的 CRUD service（POST / GET 列表 / GET 单条 / PATCH / DELETE），1 个是算法链路桥接函数 `list_for_algorithm`（B1 链路需要）。全部实现，让教师能自助维护自己的偏好，且 B1 的链路能拿到偏好数据喂给算法。

## 预期成果

- 5 个 CRUD 端点全部可用、返回 `ApiResponse[T]`，无任何 `NotImplementedError` / `raise HTTPException(...)`。
- 第 6 个 service 函数 `list_for_algorithm(db, semester) -> list[engine.TeacherPreference]` 也实现——它是 B1 `_fetch_upstream_data` 的依赖，不实现链路就跑不通。
- 权限正确：
  - 教师 / 管理员可调写接口；service 层强制比对 `current_user.user_id == preference.teacher_id`——访问别人的偏好抛 `BizException(BizCode.PERMISSION_DENIED, ...)`。
  - 学生角色一律 403（路由层 `require_teacher_or_admin` 已拦）。
- 列表接口（教师本人视角）支持按 `semester` 筛选 + `skip/limit` 分页（路由签名已定）。
- `is_negative` 字段语义保留——A1 算法会消费它。
- 在 `tests/unit/` 加一个 `test_teacher_preferences.py`：覆盖正常增删改查、跨教师访问拒绝、学生 403、参数校验、重复条目处理（至少 8 个用例）。

## 注意点

- "教师只能改自己的"在 service 层强制：每个写/读单条函数都先比对 `current_user.user_id == row.teacher_id`，不等抛 `PERMISSION_DENIED`。路由层 `require_teacher_or_admin` 只过角色，不够。
- `list_for_algorithm` 的返回类型是 `engine.TeacherPreference` **dataclass**（不是 ORM 对象，也不是 schema），桥接给算法层用——签名和 docstring 已写好，照填即可。
- 重复偏好的处理策略（同教师 + 同条件多次写入）：可以拒绝（抛业务异常），也可以覆盖——**选一种并写进 PR 描述**让 A1 知道。
- `is_negative=True` 当前由 A1 决定算软扣分还是硬约束，你这边只要保证字段能存能查。
- 列表接口的筛选条件要参数化，不要拼字符串，用 SQLAlchemy 的 `where()` 链式条件。
- 数据库会话和事务：照 `classrooms.py` 的现有写法即可——`commit/refresh` 写在 service 层。
- 现 scaffold 不支持"管理员列出所有人的偏好"——需求文档里提到的"管理员审批 / 优先级"不在本卡范围（model 也没相应字段），后期如需另开 issue。
