## 需求

`app/services/schedule_service.py` 的 `get_teacher_timetable()` 当前抛 `NotImplementedError`，实现它：按教师 ID 查到该教师在指定学期 + 可选单个教学周的课表，正确处理 `week_parity`（ALL / ODD / EVEN）对该周是否生效的判断。

## 预期成果

- `GET /api/v1/schedule/teachers/{teacher_id}/timetable?semester=...&week=N` 可用（`week` 可选，1–16）。
- 返回 `TeacherTimetableOut`（已定义在 `app/schemas/schedule.py`），其中 `entries` 是该教师在筛选范围内的条目列表。
- `week` 切片规则正确：
  - `entry.week_start <= week <= entry.week_end`
  - 且 `week_parity` 满足：`ALL`=任意；`ODD`=week 为奇数；`EVEN`=week 为偶数
- 不传 `week` 时返回该教师整学期（1–16 周）的所有条目，不做奇偶过滤，由调用方自行解读。
- 在 `tests/unit/` 加 timetable 单测：覆盖 ALL/ODD/EVEN 三种 parity 的切片正误、不传 `week` 返回整学期、跨学期边界、空结果、不同角色调用的行为。

## 注意点

- 当前路由用的是 `get_current_user`（任意登录用户都能查），权限"谁能看谁的课表"由你在 service 里决定并写进 PR 描述：建议 ADMIN 任意查、TEACHER 只查自己、STUDENT 拒绝；不等抛 `PERMISSION_DENIED`。
- `teacher_ids` 是 JSON 数组（合上课时可能多教师），过滤教师时要做"成员匹配"，**不能**用 `==` 直接比对——参考 `get_schedule_entries`（同文件 line 122-139）的 Python 侧过滤模板。
- 不传 `week` 的整学期视图：原样返回条目即可，**不要**在后端把奇偶周展开成 16 周的具体列表（那是另一种契约）。前端若需要"第 N 周"视图就再调一次带 `week=N`。
- 查询要走索引：`schedule_entries` 上目前没有 `teacher_id` 索引（model 文件没声明），如要加先在 PR 描述里说清，避免改字段触发 `make reset`。
- 性能：尽量用一次 SQL 拿全部数据，不要循环 N+1。
