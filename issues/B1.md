## 需求

把 `app/tasks/scheduler_tasks.py` 中两个空函数实现完整，让"前端点触发 → Celery 异步跑 → 数据库出结果 → 前端轮询拿到 SUCCESS"这条链路在真实数据下能端到端跑通。

## 预期成果

- `_fetch_upstream_data()` 不再抛 `NotImplementedError`：从基础信息服务（HTTP 调用）拉课程、教师、教室、教师偏好，转换成算法所需的 `list[CourseInput]` / `list[ClassroomInput]` / `list[TeacherPreference]`。
- 上游不可用时有清晰的 fallback 策略（开发期可走本地 stub，生产抛 `BizCode.UPSTREAM_FETCH_FAILED`）。
- `_save_results()` 不再 `pass`：把 `list[ScheduleResult]` 批量落 `schedule_entries` 表；更新 `ScheduleTask.status` 为 SUCCESS / PARTIAL（unscheduled 非空时）；记录 unscheduled 课程列表到 task 元数据。
- **任务全流程状态机要写全**：当前 `run_auto_schedule` 异常路径只动 Celery state、不动 DB——本卡要补齐：开始时 PENDING→RUNNING、成功 SUCCESS/PARTIAL、异常路径 FAILED 并写 `error_msg`。
- 整段任务包在事务里——save 失败要回滚，task 状态正确推到 FAILED。
- `make test-integration` 中原本 skip 的全链路用例至少跑通 trigger→poll→SUCCESS→查 entries 一条主路径。

## 注意点

- 上游字段命名很可能和算法 dataclass 不一致——映射逻辑要单独函数封装，方便单测。
- `_save_results` 要批量插入而不是循环单插，否则全校规模会很慢。
- 状态机不能跳级：PENDING→RUNNING→{SUCCESS|FAILED|PARTIAL}，不要从 PENDING 直接跳 SUCCESS。
- 进度上报用 `self.update_state(state="PROGRESS", meta={"progress": N, "message": "..."})`——前端轮询拿这个。
- `_save_results` 里 `task_id` 这个变量名指的是 **Celery 任务 ID（字符串）**，不是 `ScheduleTask.id`（int 外键）；用前者反查 `ScheduleTask` 行拿后者，写 `ScheduleEntry.task_id` 时用后者。
- A1 的算法可能还在写，开发时算法部分可以先用现成 stub `run_schedule()` 跑通链路；接口合入顺序：先合 B1（链路通），再合 A1（替换 stub）。
- `_fetch_upstream_data` 的 stub fallback **不要**在生产代码路径默认启用——用环境变量或 settings 开关控制，避免线上掩盖真实失败。
- 依赖 B2 的 `list_for_algorithm` —— 同 sprint 推进时先确认 B2 这一函数的 PR 状态。
